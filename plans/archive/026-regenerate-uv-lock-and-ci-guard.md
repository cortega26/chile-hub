## Why this matters

`uv.lock` is out of sync with `pyproject.toml`. `pyproject.toml` pins
`python-semantic-release==10.6.1` (line 65), but `uv.lock` records the specifier
`==10.5.3` and resolves that version (`uv.lock:534`). Running `uv lock --locked`
**fails** ("The lockfile needs to be updated"). This breaks the project's own stated
reproducibility guarantee — `AGENTS.md §10`: *"Las dependencias se fijan en
`pyproject.toml` y se reproducen exactamente vía `uv.lock`."* A contributor or CI step
running `uv sync` today silently resolves a **different** semantic‑release than declared,
and there is no gate that catches lock drift. The bump commit that introduced this
(`23ac012`) edited the manifest without regenerating the lock.

## Current state

- `pyproject.toml:65` — `"python-semantic-release==10.6.1",` (in the `dev` extra).
- `uv.lock:534` — `{ name = "python-semantic-release", marker = "extra == 'dev'", specifier = "==10.5.3" }`
  and the resolved package entry pins `version = "10.5.3"`.
- `.github/workflows/pipeline-check.yml` — the `quality` job (lines 30‑55) runs only ruff;
  no job runs `uv lock --locked` / `uv sync --locked`, so drift is never detected in CI.
- `Makefile` uses `uv sync --extra pipeline --extra dev` in `bootstrap` (line 74) — non‑locked,
  so it masks the drift locally too.

## Commands you will need

| Purpose | Command | Expected on success |
|---|---|---|
| Detect drift (before) | `uv lock --locked` | exits **non‑zero** now (the bug) |
| Regenerate lock | `uv lock` | exit 0; `uv.lock` updated |
| Verify in sync (after) | `uv lock --locked` | exit 0 |
| Sync check | `uv sync --extra pipeline --extra dev --locked` | exit 0 |

(Requires `uv` on PATH. If unavailable, STOP — do not hand‑edit `uv.lock`.)

## Scope

**In scope**:

**Out of scope**:
- `pyproject.toml` — its pins are the source of truth; do NOT downgrade the pin to match the
  stale lock. The lock must move to match the manifest, not vice versa.
- Any dependency version choice — this plan only reconciles lock↔manifest, it does not bump anything
  on purpose. If `uv lock` pulls incidental transitive updates, that is expected; review the diff.

## Steps

### Step 1: Confirm the drift, then regenerate the lock

```
uv lock --locked   # expect: non-zero, "lockfile needs to be updated"
uv lock            # regenerates uv.lock
uv lock --locked   # expect: exit 0
```

Inspect the diff (`git diff uv.lock | head -80`): the intended change is
`python-semantic-release` moving `10.5.3` → `10.6.1` (plus any transitive deps that version
requires). If the diff is enormous or downgrades core runtime libs (polars/duckdb/pyarrow),
STOP and report — that suggests the lock was generated against a different index or Python.

**Verify**: `uv lock --locked` → exit 0.

### Step 2: Add a lockfile‑drift guard to CI

In `.github/workflows/pipeline-check.yml`, add a step to the existing `quality` job (after the
"Set up uv" step, alongside the ruff steps) that fails when the lock is stale:

```yaml
      - name: Verify uv.lock is in sync with pyproject.toml
        run: uv lock --locked
```

The `quality` job already sets up `uv` (`astral-sh/setup-uv`), so no extra setup is needed.

**Verify**: `grep -n "uv lock --locked" .github/workflows/pipeline-check.yml` → shows the new step.

### Step 3: Confirm a locked sync works end‑to‑end

```
uv sync --extra pipeline --extra dev --locked
```

**Verify**: exit 0 (no "lockfile needs to be updated").
